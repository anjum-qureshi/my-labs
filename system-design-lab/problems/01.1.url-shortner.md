# 1. Problem Statement
Design a URL shortening service that converts long URLs into compact, shareable URLs and redirects users to the original destination with low latency and high availability.
---

# 2. Functional Requirements
  Users can create a short code or short url for a given url.
  Users are redirected to actual urls when given a short code url.
  configurable expiration
  optional custom alias
---

# 3. Non-Functional Requirements
  - low latency 100ms redirect
  - high throughout for read/redirect
  - Highly available
  - Read heavy system
  - Durable storage
  - Eventual consistency Acceptable
  - Low infrastructure cost
  - Logging/tracings
  - Fault tolerant

Generate the short-code
Approach 1: Hash the long url and store the hash as a shortcode. Hashing doesn't really make the url short the hash itself is long.
cutting down the hash we will reduce the length to n, increases the collision. so before saving the short against any url should be validated if it already given to some other url.
if its already taken regenerate until we find a unique short code. also convert shortcode to base62 to reduce the size and also avoid having =,/,? as they are special characters in a url. or use url-base-64 encoding.

Approach 2: 
 Using snowflake mechanism, which is unqiue 64-bit id , mix of timestamp, machine id which is local sequence would be unqiue.

Approach 3:
Decide another mechanism generating a fixed length of a series of base62 strings from 00000000 to zzzzzz, so whenever a new url request comes we just assign it not next short code rather than worrying of collision.

# 12. Scaling

Question

How do I remove ONE bottleneck?

Today we had the database as a heavy bottleneck (hot spot for reading queries).
1. Database become hotspot for 100M reads/Day as mentioned with cache-aside strategy.
  Introduced redis cache which stores the copy the same database record with a ttl. So every 100 queries, 10 go to database for cache miss.
  Using postgresql for our storage, as we have structed dataset, which requires concurrent writes and read operations with high availability.


2. When the reads increased from 1B/day the redis is performing well, database queries are performing fine. Find the request is spending more time in application to store the traces and analytics. Introduce kafka to store analytics asynchronous as its not required for users.

3. Writes increased from 1M/day to 10M/day won't have much impact on our achitecture as it just 120 writes/sec postgresql can handle that. If write go to 2000 writes/sec understand where the write is taking time at the connection pooling, write latency. Introduce read and write replicas Even now if the latency doesn't reduce introduce partitioning the database based short code or id.
